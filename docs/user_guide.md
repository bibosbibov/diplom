# Руководство пользователя

Этот документ предназначен для **инженера по информационной безопасности**,
который хочет использовать систему для практической оценки критичности
уязвимостей — без погружения в детали machine learning. Если интересует
техническая архитектура, см. [docs/architecture.md](architecture.md).

## Что делает система

Вы вводите **текстовое описание уязвимости** (на русском или английском)
и **идентификатор CWE** (например, `CWE-89` для SQL injection). Система:

1. Предсказывает 12 метрик базового вектора **CVSS v4.0**;
2. Рассчитывает итоговый числовой балл (от 0,0 до 10,0);
3. Определяет уровень критичности (None / Low / Medium / High / Critical);
4. Возвращает уверенность модели по каждой метрике (для оценки доверия).

Время ответа — ~200 мс на CPU, что делает систему пригодной для интерактивной
работы.

## Веб-интерфейс: пошаговое руководство

После запуска `uvicorn src.api.main:app` откройте <http://localhost:8000>.

### Поля формы

| Поле | Назначение | Обязательно |
|:--|:--|:--|
| **Описание уязвимости** | Свободный текст 10–10 000 символов, рус. или англ. Чем подробнее, тем выше качество. | Да |
| **Тип уязвимости (CWE)** | Выпадающий список топ-30 наиболее частых CWE. Можно ввести любой `CWE-N` вручную. | Да |
| **EPSS** | Слайдер 0–1: оценка вероятности эксплуатации в ближайшие 30 дней. Если оставить «∅» — модель использует маркер «отсутствует». | Нет |
| **CISA KEV** | Checkbox: уязвимость есть в каталоге известно эксплуатируемых CISA. | Нет |
| **Публичный эксплойт** | Checkbox: для уязвимости опубликован эксплойт (ExploitDB). | Нет |

### Что в результате

После нажатия **«Оценить»** появится блок с тремя частями:

1. **Большая цветная плашка** с итоговым уровнем критичности (Critical/High/
   Medium/Low/None) и численным баллом 0–10.
2. **CVSS-вектор** в моноширинном шрифте, кнопка «Скопировать» для переноса
   в багтрекер или отчёт.
3. **Таблица 12 метрик** с предсказанным значением и confidence (зелёная
   подсветка — высокая уверенность > 0,8; жёлтая — средняя 0,5–0,8;
   красная — низкая < 0,5).

Под таблицей выводится:

- **«Низкая уверенность»** — список метрик, требующих ручной проверки;
- **«Время инференса»** — в миллисекундах.

> **Совет:** если несколько метрик подсвечены красным, не доверяйте
> предсказанию слепо. Сверьтесь с описанием уязвимости и спецификацией CVSS
> v4.0. Низкая уверенность — сигнал «требуется аналитик».

## Что такое CVSS v4.0

**CVSS** (Common Vulnerability Scoring System) — открытый отраслевой стандарт
от [FIRST.org](https://www.first.org/cvss/) для количественной оценки тяжести
уязвимостей ПО. Версия **4.0** опубликована в ноябре 2023 г. и пришла на смену
v3.1, расширив набор метрик:

- введены **AT** (Attack Requirements), **SC/SI/SA** (Subsequent System
  Impact — влияние на смежные системы);
- разделены impact на **уязвимую систему** (Vulnerable: VC/VI/VA) и
  **последующие системы** (Subsequent: SC/SI/SA);
- пересмотрены формулы базового балла — теперь используется lookup по
  таблице из 264 значений (MacroVector) с дополнительной интерполяцией.

Базовый вектор CVSS v4.0 имеет вид:

```
CVSS:4.0/AV:N/AC:L/AT:N/PR:N/UI:N/VC:H/VI:H/VA:H/SC:N/SI:N/SA:N/E:A
```

Итоговый балл рассчитывается по 11 обязательным метрикам + опциональной
Exploit Maturity (E).

## 12 метрик CVSS v4.0

Краткие определения из официальной [спецификации FIRST](https://www.first.org/cvss/v4.0/specification-document).

### Группа Exploitability (как уязвимость эксплуатируется)

| Метрика | Полное название | Допустимые значения | Краткое определение |
|:--|:--|:--|:--|
| **AV** | Attack Vector | N · A · L · P | Среда атаки: Network / Adjacent network / Local / Physical |
| **AC** | Attack Complexity | L · H | Требуются ли спец. условия (защита от детекции и т.п.) |
| **AT** | Attack Requirements | N · P | Зависит ли атака от внешних условий (специфика конфигурации, гонка) |
| **PR** | Privileges Required | N · L · H | Какой уровень доступа нужен атакующему до атаки |
| **UI** | User Interaction | N · P · A | Требуется ли взаимодействие пользователя (пассивное/активное) |

### Группа Impact на уязвимой системе

| Метрика | Полное название | Значения | Краткое определение |
|:--|:--|:--|:--|
| **VC** | Vulnerable: Confidentiality | H · L · N | Утечка/раскрытие данных самой уязвимой системы |
| **VI** | Vulnerable: Integrity | H · L · N | Возможность модификации данных уязвимой системы |
| **VA** | Vulnerable: Availability | H · L · N | Возможность отказа в обслуживании уязвимой системы |

### Группа Subsequent System Impact (новое в v4.0)

| Метрика | Полное название | Значения | Краткое определение |
|:--|:--|:--|:--|
| **SC** | Subsequent: Confidentiality | H · L · N | Утечка данных смежных систем (через уязвимую) |
| **SI** | Subsequent: Integrity | H · L · N | Модификация данных смежных систем |
| **SA** | Subsequent: Availability | H · L · N | DoS смежных систем |

### Threat-метрика

| Метрика | Полное название | Значения | Краткое определение |
|:--|:--|:--|:--|
| **E** | Exploit Maturity | A · P · U | Зрелость публичного эксплойта: Attacked / Proof-of-Concept / Unreported |

## Уровни критичности

Маппинг численного балла на качественный уровень (по спецификации FIRST):

| Severity | Балл | Цвет в UI | Практический смысл |
|:--|:--|:--|:--|
| **Critical** | 9,0 – 10,0 | Красный | Немедленное реагирование, изоляция, патч ASAP |
| **High** | 7,0 – 8,9 | Оранжевый | Высокий приоритет, патч в течение нескольких суток |
| **Medium** | 4,0 – 6,9 | Жёлтый | Плановый патч в очередном цикле обновлений |
| **Low** | 0,1 – 3,9 | Зелёный | Низкий приоритет, наблюдение |
| **None** | 0,0 | Серый | Уязвимость нерелевантна (например, корректные ограничения уже есть) |

## Сценарии использования

### Сценарий 1. Triage уязвимости из багтрекера компании

**Контекст.** Junior-инженер по ИБ получает в Jira / GLPI / OTRS тикет от
DevOps-команды: «обнаружена возможность SQL-инъекции в форме логина
admin-панели». CVE ещё не присвоен — публикация в NVD будет позже. Нужно
расставить приоритет.

**Действия:**

1. Открыть веб-интерфейс системы.
2. Скопировать описание из тикета в поле «Описание».
3. Выбрать в селекторе **CWE-89 (SQL Injection)**.
4. Оставить EPSS пустым (нет CVE → нет EPSS), KEV/exploit не отмечать.
5. Нажать «Оценить».

**Результат.** Система выдаёт `CVSS:4.0/.../VC:H/VI:H/VA:H/.../E:U` со скором
~8,2 (High). Аналитик получает количественный ориентир для SLA и аргумент
для эскалации.

### Сценарий 2. Быстрая оценка известного CVE

**Контекст.** В команде обсуждают свежий CVE из NVD. У него уже есть EPSS,
есть пометка «попал в CISA KEV». Хочется получить вектор CVSS v4.0 (NVD пока
содержит только v3.1).

**Действия:**

1. Скопировать описание из NVD в поле «Описание».
2. Указать CWE из карточки NVD.
3. Установить **EPSS** на полученное значение, отметить **CISA KEV**.
4. Если в ExploitDB есть рабочий PoC — отметить **Публичный эксплойт**.
5. Получить вектор CVSS v4.0 и обоснование уровня критичности.

### Сценарий 3. Batch-обработка списка уязвимостей

**Контекст.** В компании есть outstanding-список из ~200 уязвимостей,
обнаруженных при сканировании. Часть имеет CVE, часть — нет. Нужно отсортировать
по критичности.

**Действия:**

```bash
# Подготовить CSV с колонками description, cwe_id, epss, kev, exploit
python -m src.inference.cli batch-predict vulns.csv vulns_scored.csv

# Открыть результат и отсортировать по полю score
```

Альтернативно — обращаться к `POST /predict/batch` (до 100 элементов за
запрос). Полное описание — [docs/api.md](api.md).

## Ограничения системы

> Это **decision-support tool**, а не финальный судья. Ниже — честный
> список ограничений, которые важно учитывать при практическом применении.

### 1. Качество зависит от описания

Модель работает только с текстом. Если описание короткое («buffer overflow
in handle()») или предельно общее («there is a vulnerability in component
X»), точность падает резко. Хорошее описание содержит:

- **что именно** делает злоумышленник (отправляет HTTP-запрос, скармливает
  файл и т.п.);
- **где** проявляется (компонент, функция, API endpoint);
- **к чему** приводит (DoS, RCE, утечка данных, эскалация).

### 2. Лучше работает на типичных шаблонах

Качественные показатели (`per-CWE` анализ, [reports/CHAPTER3_DRAFT.md](../reports/CHAPTER3_DRAFT.md)):

| Тип уязвимости | CWE | Среднее число правильных метрик |
|:--|:--|--:|
| SQL Injection | CWE-89 | 10,4 / 11 (отлично) |
| Buffer Overflow | CWE-119/121 | 9,5 / 11 (хорошо) |
| XSS | CWE-79 | 8,9 / 11 (нормально) |
| Path Traversal | CWE-22 | 7,8 / 11 (слабее) |
| Improper Access Control | CWE-284 | 7,7 / 11 (слабее) |

Чем чётче лексический «фингерпринт» уязвимости в описании, тем выше точность.

### 3. Хуже на редких типах и нестандартных описаниях

Уязвимости из категорий «UI Misrepresentation», «Authorization Bypass»
(CWE-862, CWE-863), сложные многоступенчатые атаки и race conditions —
зоны, где модель ошибается чаще. В этих случаях ориентируйтесь на confidence:
если ≥3 метрик подсвечены красным, **обязательно** проверить вручную.

### 4. Не учитывает контекст инфраструктуры

CVSS Base — это «природа уязвимости в вакууме». Реальная критичность зависит
от того, **где** уязвимая система установлена, какие данные через неё проходят,
есть ли компенсирующие меры. Это поле Environmental Score (раздел 7.3
спецификации CVSS v4.0) — система **его не считает**, требуется
самостоятельная корректировка аналитиком.

### 5. Калибровка confidence неидеальна

Средний confidence на верных предсказаниях — 0,905, на неверных — 0,725.
Это означает, что **низкий confidence — почти всегда индикатор ошибки**,
но **высокий confidence не гарантирует правильность**. Из 11 664 предсказаний
на test set 7 431 имели confidence > 0,9 — часть из них ошибочны
(см. `reports/error_analysis/high_confidence_errors.csv`).

### 6. Не заменяет полный аудит безопасности

Система предназначена для **первичной триажной оценки**: расставить
приоритеты в большом списке, дать количественную метрику для отчётности,
сократить время на рутину. Для критичных уязвимостей в production-системах
**обязательна** ручная экспертная оценка.
